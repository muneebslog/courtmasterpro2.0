# ScoreMaster — Full Application Specification
> Badminton Tournament Management System for Federation Use
> Reference document for developers, AI builders, and federation admins.

---

## 1. Purpose

ScoreMaster replaces the paper-based tournament tracking system used by the badminton federation. It digitises every layer of a tournament — from creation to final result — and provides live scores to the public in real time.

It is not a draw-generation tool. The federation makes draws on paper. ScoreMaster receives those draws and manages everything from that point forward.

---

## 2. User Roles

| Role | Access | Login |
|---|---|---|
| Admin | Full control — create tournament, events, stages, draws, corrections | Unique login, 1 admin = 1 tournament |
| Umpire | Start matches, enter scores, log occurrences | One shared umpire account |
| Public | Read-only — live scores, bracket, match history | No login required |

---

## 3. Data Hierarchy

Every piece of data in the system lives in this hierarchy. Nothing exists outside it.

```
Tournament
└── Event (e.g. Men's Singles U16, Mixed Doubles Open)
    └── Stage (e.g. Round of 64, Quarter Final, Semi Final, Final)
        └── Tie  ← only exists in Team events
        │   └── Match (S1, D1, S2, D2, S3 — fixed order)
        │       └── Game (Game 1, Game 2, Game 3)
        │           └── Event Log (every point, occurrence, correction)
        └── Match  ← for Individual events (Singles / Doubles)
            └── Game
                └── Event Log
```

---

## 4. Tournament

### 4.1 Creation
Admin logs in and creates one tournament. One admin account = one active tournament at a time.

Fields:
- Tournament name
- Venue
- Start date / End date
- Status: `Draft` → `Active` → `Completed`

### 4.2 Rules
- Tournament cannot be deleted once any match has started.
- Admin can edit tournament name, venue, and dates at any time before completion.

---

## 5. Events

### 5.1 What is an event
An event is a category of competition within a tournament. Each event runs independently — stages in one event do not affect stages in another.

### 5.2 Event fields
- Name (e.g. Men's Singles U16, Women's Doubles Open, Team Event U19)
- Type: `Singles` | `Doubles` | `Team`
- Age group / category (e.g. U16, U19, Open) — optional label

### 5.3 Event types behaviour
| Type | Match structure | Players per side |
|---|---|---|
| Singles | 1 player vs 1 player | 1 |
| Doubles | 1 pair vs 1 pair | 2 |
| Team | Team vs Team (Tie = 5 matches) | Roster of players |

---

## 6. Stages

### 6.1 What is a stage
A stage is a round within an event. Examples: Round of 64, Round of 32, Quarter Final, Semi Final, Final.

### 6.2 Stage fields
- Stage name (admin defines it — e.g. "Round of 64", "QF", "Final")
- Best of: `1` | `3` | `5` — applies to all matches in this stage
- Number of matches in a tie (team events) — always 5, fixed order: S1, D1, S2, D2, S3

### 6.3 Stage progression
- Admin creates the first stage and enters the draw.
- When all matches in a stage are completed, the system **automatically creates the next stage** and populates it with the winners.
- Admin does not need to manually create subsequent stages.
- Auto-advance happens only when all matches in the current stage are in a terminal state: `Completed`, `Walkover`, `Retired`, or `Not Required`.

### 6.4 Multiple stages running simultaneously
Events run independently. It is valid for Men's Singles to be at Quarter Final stage while Women's Doubles is still at Round of 32. Stages within the same event are sequential.

---

## 7. Draw Entry

When admin creates a stage, they enter the draw — who plays who. The system creates matches from this input.

### 7.1 Singles / Doubles draw entry
Admin sees a form with rows. Each row is one match.

```
Match 1:  [Player A name]  vs  [Player B name]
Match 2:  [Player C name]  vs  [Player D name]
[ + Add match ]
```

For doubles, each side has two name fields:
```
Match 1:  [Player A1] + [Player A2]  vs  [Player B1] + [Player B2]
```

- Leaving one side blank = **BYE**. That player auto-advances, no match is created.
- Admin submits the whole form at once. All matches are created in one action.

### 7.2 Team event draw entry
Admin enters team names only:
```
Tie 1:  [Team A]  vs  [Team B]
Tie 2:  [Team C]  vs  [Team D]
[ + Add tie ]
```

After saving, admin clicks into each team to **add players to the roster**. The 5 matches inside each tie (S1, D1, S2, D2, S3) are auto-generated by the system in fixed order. Admin does not create them manually.

### 7.3 Player assignment in team ties
Players are **not** pre-assigned to specific matches inside a tie. Assignment happens at match start time by the umpire — they select who is playing from the team roster at that moment.

**Player eligibility rule per tie:**
- One player can play a maximum of **1 singles match** and **1 doubles match** per tie.
- The system enforces this on the umpire's player selection screen — players who have already played both their singles and doubles are greyed out.

---

## 8. Match Structure

### 8.1 Individual event match
- Player/pair A vs Player/pair B
- Played as best of 1, 3, or 5 games (defined at stage level)
- Each game played to 21 points
- Deuce at 20–20: must win by 2, maximum 30–29
- First side to win the required number of games wins the match

### 8.2 Team event tie
- Team A vs Team B
- Contains exactly 5 matches in fixed order: S1, D1, S2, D2, S3
- First team to win 3 matches wins the tie
- If a team reaches 3 wins before all 5 matches are played, remaining matches are marked **Not Required** and never started
- Matches inside a tie are played **sequentially** — not in parallel

### 8.3 Match statuses
| Status | Meaning |
|---|---|
| `Pending` | Created, not yet started |
| `In Progress` | Umpire has started the match |
| `Completed` | Match finished normally, result confirmed |
| `Retired` | Player retired or was injured mid-match |
| `Walkover` | Player did not show up before match started |
| `Not Required` | Team tie decided early, match never played |

---

## 9. Event Log (The Timeline)

Every match has a full event log. This is the core data model for scoring. The score is not stored as a number — it is the **sum of all point events** in the log.

### 9.1 Event types
| Event type | When | Contains |
|---|---|---|
| `match_started` | Umpire hits start | umpire name, service judge name, court, timestamp |
| `point` | Umpire taps a side | side scored, score after point, server, timestamp |
| `undo` | Umpire undoes last point | previous score restored, reason, timestamp |
| `occurrence` | Umpire logs an event | type, player, score at time, timestamp |
| `game_ended` | System detects game won | winner, final game score, timestamp |
| `bulk_score_entry` | Offline/missed game | final game score only, no point-by-point, flag: `offline_entry: true` |
| `match_ended` | Umpire confirms end | winner, final score, timestamp |
| `match_reset` | Admin resets match | admin name, reason, timestamp |
| `player_edit` | Admin edits player name | old name, new name, admin name, timestamp |
| `score_correction` | Admin corrects score | old score, new score, admin name, reason, timestamp |

### 9.2 Why this model
- Full audit trail of every action
- Score can be reconstructed at any point in time
- Offline/missed games can be bulk-entered without corrupting the timeline
- Corrections are transparent — nothing is silently overwritten

## 10. Scoring

### 10.1 Live tap mode
Umpire taps Side A or Side B button. System adds a `point` event to the log and updates the live score instantly. Public sees the update in real time.

### 10.2 Bulk entry mode (offline fallback)
Used when internet was lost or umpire missed entering a game live. Umpire enters the final score for a game (e.g. 21–15). System creates one `bulk_score_entry` event. No point-by-point timeline exists for this game. This is clearly flagged on the match timeline view.

### 10.3 Both modes can coexist in the same match
Game 1 may have a full point-by-point timeline. Game 2 may only have a bulk entry. Both are valid.

### 10.4 Deuce logic
- At 20–20: game continues, no automatic end
- Side must reach 2 point lead
- Hard cap: game ends at 30–29 regardless

### 10.5 Game end
System auto-detects when a side wins a game. A prompt appears: "[Side A] won Game 1 — 21–15. Start Game 2?" Umpire taps confirm. Sides swap automatically per badminton rules.

### 10.6 Match end
When one side wins the required number of games, system prompts: "[Side A] won the match — 2–1. Confirm result?" Umpire confirms. Result is locked. Winner auto-advances in bracket.

---

## 11. Occurrences

Occurrences are logged at any point during a match. Each occurrence is a timestamped event in the match log with the score at the time of the occurrence.

### 11.1 Occurrence types
| Type | Effect on match |
|---|---|
| Shuttle change | Informational only, no score effect |
| Yellow card | Informational, logged against player |
| Red card | Point deducted from player's side OR disqualification — federation defines which rule applies |
| Injury | Informational — umpire decides whether to continue or retire |
| Retirement | Terminates the match — opponent wins |
| Walkover | Before match starts — opponent wins, no score |

### 11.2 Retirement flow
1. Umpire logs occurrence type `Injury`
2. If player cannot continue, umpire taps "Retire player"
3. System saves all scores up to that point
4. Match status set to `Retired`
5. Opponent is declared winner
6. Winner auto-advances in bracket

### 11.3 Walkover flow
1. Umpire opens match on pre-start screen
2. Player has not shown up
3. Umpire taps "Walkover"
4. No score is entered
5. Match status set to `Walkover`
6. Opponent auto-advances

---

## 12. Undo

- Umpire can undo the last point at any time during a live match
- Undo is immediate — live score updates instantly for public
- Every undo is logged as an `undo` event in the timeline — it is never silently removed
- Undo shows a confirmation: "Undo last point? [Side A] 11 → 10" — prevents accidental taps
- Only the last point can be undone (one level deep) — no multi-level undo

## 13. Umpire Flow (Step by Step)

### Step 1 — Login
Umpire logs in using the shared umpire account.

### Step 2 — Match list
Sees all matches across all events. Filtered by status: Pending / In Progress / Completed. Finds their assigned match.

### Step 3 — Open match
Sees match info: event name, stage, Side A vs Side B, court (if assigned by admin), best of.

### Step 4 — Pre-start setup
Before scoring begins, umpire fills in:
- Umpire name (free text)
- Service judge name (free text)
- Court number (free text)
- For team event: select players for this specific match from the team roster
  - Singles match: select 1 player from each team
  - Doubles match: select 2 players from each team
  - Eligibility enforced: max 1 single + 1 double per player per tie

### Step 5 — Start match
Umpire taps "Start Match". Match status changes to `In Progress`. Timeline begins.

### Step 6 — Live scoring
Main scoring screen shows:
- Side A name (large, left half of screen)
- Side B name (large, right half of screen)
- Current game score (very large, centre)
- Games won count (e.g. 1–0)
- Undo button (visible but not prominent)
- Occurrence button (accessible but not in the scoring area)
- Umpire taps left half = Side A scored. Taps right half = Side B scored.

### Step 7 — Log occurrence (any time)
Umpire opens occurrence panel, selects type, selects player if applicable, confirms. Returns to scoring screen.

### Step 8 — Game ends
System auto-detects. Shows prompt with game result. Umpire confirms. Next game begins.

### Step 9 — Match ends
System auto-detects. Shows match result prompt. Umpire confirms. Result locked. Timeline sealed.

### Step 10 — Winner advances
System auto-advances winner into next stage bracket slot.

## 14. Admin Flow (Step by Step)

### Step 1 — Login
Admin logs in with unique credentials.

### Step 2 — Tournament dashboard
Sees the active tournament. Overview of all events, stages, and match statuses.

### Step 3 — Create tournament
One-time setup: name, venue, dates.

### Step 4 — Create event
Name, type (Singles / Doubles / Team), age group.

### Step 5 — Create stage
Stage name, best of (1/3/5).

### Step 6 — Enter draw
Fill in the draw form (see Section 7). Submit. Matches are created.

### Step 7 — Monitor
Watch the bracket and match list. See live scores, match statuses, and results as they come in.

### Step 8 — Team roster management
For team events: click into any team, add/edit players in the roster at any time before the relevant match starts.

### Step 9 — Score corrections (when needed)
See Section 15.

## 15. Edit, Reset, and Correction Rules

These rules are based on real operational experience. Strict immutability causes more problems than it solves. These rules balance data integrity with practical flexibility.

### 15.1 Before match starts (`Pending`)
| Action | Allowed | Notes |
|---|---|---|
| Edit player names | ✅ Yes | Admin only |
| Edit team roster | ✅ Yes | Admin only |
| Delete match | ✅ Yes | Only if no subsequent match in bracket has started |
| Change best of | ✅ Yes | Admin only, stage-level setting |

### 15.2 Match in progress (`In Progress`)
| Action | Allowed | Notes |
|---|---|---|
| Edit player names | ✅ Yes | Admin only — name corrections happen mid-tournament |
| Edit score | ❌ No | Umpire uses undo on tablet |
| Delete match | ❌ No | Cannot delete a live match |

### 15.3 Match completed (`Completed`)
| Action | Allowed | Notes |
|---|---|---|
| Edit player names | ✅ Yes | Admin only — registration corrections |
| Reset match | ✅ Yes | Admin only — see reset rules below |
| Delete match | ❌ No | |
| Direct score edit | ❌ No | Must use reset flow |

### 15.4 Reset match rules
Reset wipes the entire score and timeline, returning the match to `Pending` status so umpire can re-enter scores.

**Reset is blocked if:** The winner of this match has already **started** their next match in the bracket. If the next match is still `Pending`, reset is allowed.

**Reset flow:**
1. Admin clicks "Reset Match"
2. System checks: has the winner's next match started? If yes — blocked with message.
3. If allowed — admin enters reason (required)
4. Match returns to `Pending`
5. Next stage bracket slot is cleared — winner removed
6. All events logged: `match_reset` with admin name, timestamp, reason

### 15.5 Score correction (without full reset)
For minor corrections where the winner is not in dispute (e.g. score was entered as 21–15 but should be 15–21 — same winner, just the sides were swapped):

1. Admin opens match
2. Selects "Correct score"
3. Edits the score values
4. System logs `score_correction` event with old/new values, admin name, timestamp
5. Winner does not change — bracket is unaffected

**If the winner would change:** Admin must use full reset instead.

**If the next stage match has already started:** Correction is blocked. Admin must resolve manually (database-level only, outside the app).

### 15.6 Audit trail
Every action in sections 15.1–15.5 is logged with: admin name, timestamp, what changed, and reason. This log is visible to admin only — not on the public view.

## 16. Public View

No login required. Accessible by anyone with the tournament URL.

### 16.1 Pages
| Page | Content |
|---|---|
| Tournament home | Tournament name, venue, dates, list of events |
| Event page | All stages for this event |
| Bracket view | Visual bracket showing all matches, results, and current live matches |
| Live match | Live score updated in real time, current game, games won |
| Match detail | Full result, game-by-game scores |
| Match timeline | Full event log — every point, occurrence, and correction — with timestamps |

### 16.2 Live update behaviour
- Scores update in real time during a live match
- If connection to server is lost (umpire offline), public screen shows "Score update paused"
- When connection restores, score updates resume

### 16.3 What public does NOT see
- Admin audit log
- Correction history
- Umpire/service judge names (optional — federation can decide)
- Offline entry flags (optional — federation can decide)

## 17. Offline / Connection Loss Handling

### 17.1 Umpire tablet loses connection
- Tablet queues all point events locally
- Umpire continues scoring normally
- When connection restores, queued events sync to server automatically
- Public score feed is paused during disconnect, resumes on reconnect

### 17.2 Full game missed (server was down)
- Umpire uses bulk entry mode when back online
- Enters final game score manually
- System creates a `bulk_score_entry` event — flagged as offline entry
- No point-by-point timeline for that game

### 17.3 These two modes can coexist in the same match
Any combination of live-tap games and bulk-entry games in the same match is valid.

---

## 18. BYE Handling

When a player/team has no opponent in a round (odd number of participants):

- Admin leaves the opponent side blank when entering the draw
- System marks that slot as `BYE`
- No match is created
- Player auto-advances to next stage immediately
- Bracket shows "BYE" in that slot

## 20. Key Business Rules Summary

| Rule | Detail |
|---|---|
| 1 admin = 1 tournament | One admin account manages one active tournament |
| Events are independent | Stages in different events run in parallel without affecting each other |
| Stages are sequential | Within one event, stages run one at a time |
| Auto-advance | System auto-creates next stage when current stage fully complete |
| Team tie = 5 matches fixed | S1, D1, S2, D2, S3 — always in this order |
| Tie won at 3 | Remaining matches become Not Required |
| Tie matches sequential | No parallel matches within one tie |
| Player per tie max | 1 player can play max 1 single + 1 double per tie |
| BYE = blank opponent | Auto-advance, no match created |
| Deuce cap | 20–20 → win by 2 → hard cap 30–29 |
| Undo is logged | Every undo creates an audit event — nothing silently deleted |
| Walkover = before start | Player no-show before match starts — no score |
| Retirement = mid-match | Player can't continue — partial score saved, opponent wins |
| Reset blocked if next started | Cannot reset a match if winner's next match is In Progress |
| Correction blocked if next started | Cannot correct score if winner has already played next match |
| All edits are logged | Admin name, timestamp, old value, new value, reason |

## 21. Data Models (Summary)

### Tournament
```
id, name, venue, start_date, end_date, status, admin_id, created_at
```

### Event
```
id, tournament_id, name, type (singles|doubles|team), category, created_at
```

### Stage
```
id, event_id, name, best_of (1|3|5), status, order_index, created_at
```

### Team (team events only)
```
id, stage_id, name, created_at
```

### TeamPlayer
```
id, team_id, player_name, created_at
```

### Tie (team events only)
```
id, stage_id, team_a_id, team_b_id, winner_team_id, status, created_at
```

### Match
```
id, stage_id, tie_id (nullable), side_a_label, side_b_label,
match_order (S1|D1|S2|D2|S3 for team, null for individual),
status, winner_side, umpire_name, service_judge_name, court,
started_at, ended_at, created_at
```

### MatchPlayer
```
id, match_id, side (a|b), player_name, position (1|2 for doubles)
```

### Game
```
id, match_id, game_number, score_a, score_b, winner_side,
entry_mode (live|bulk), started_at, ended_at
```

### MatchEvent (the timeline log)
```
id, match_id, game_id (nullable), event_type, side (nullable),
player_name (nullable), score_a_at_time, score_b_at_time,
notes, created_by (umpire|admin), created_at
```

## 22. Glossary

| Term | Meaning |
|---|---|
| Tournament | The top-level competition being managed |
| Event | A category within the tournament (e.g. Men's Singles U16) |
| Stage | A round within an event (e.g. Quarter Final) |
| Tie | A team vs team contest made up of 5 matches |
| Match | A singles or doubles contest between two sides |
| Game | One set within a match, played to 21 points |
| Draw | The list of who plays who in a stage — entered by admin |
| BYE | An empty opponent slot — player auto-advances |
| Walkover | Opponent did not show — awarded without play |
| Retirement | Player withdrew mid-match due to injury |
| Occurrence | Any notable event during a match (card, shuttle change, injury) |
| Timeline | The full chronological event log of a match |
| Bulk entry | Entering a final game score without point-by-point data |
| Reset | Wiping a completed match back to Pending for re-entry |
| Not Required | A match inside a tie that was never played because tie was decided early |
